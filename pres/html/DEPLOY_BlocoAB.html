BLOCO A — Deploy das Presenças

Passo 1 — Setup da Planilha

ℹ️
A aba PRESENCAS precisa de 3 colunas novas: ENTIDADE, TIPO, TIMESTAMP. A função _setupPresencasExt() faz isso automaticamente.
  1. 1
    Abrir o Google Apps Script
    Acesse o GAS do projeto CMS-MOC. Menu: Extensões → Apps Script no Google Sheets da planilha.
  2. 2
    Colar o Code_BlocoA_presencas.gs
    Abra Code.gs. Localize o comentário // ══ UTILITÁRIOS e cole o conteúdo do arquivo ACIMA dessa linha (antes da função _temAcesso).
  3. 3
    Adicionar os 3 cases no doGet()
    No switch do doGet(), localize case 'getPresencas': e adicione logo após:
case 'getReuniaoCurrent':    result = handleGetReuniaoCurrent(params);   break;
case 'marcarPresenca':       result = handleMarcarPresenca(params);       break;
case 'registrarVisitante':   result = handleRegistrarVisitante(params);  break;
  1. 4
    Adicionar wrappers na seção WRAPPERS
    Localize a seção com os wrappers e adicione:
function getReuniaoCurrent(p)    { return handleGetReuniaoCurrent(p || {}); }
function marcarPresenca(p)       { return handleMarcarPresenca(p || {}); }
function registrarVisitante(p)   { return handleRegistrarVisitante(p || {}); }
  1. 5
    Executar _setupPresencasExt()
    No GAS, selecione a função _setupPresencasExt no dropdown e clique em ▶ Executar. Confirme a autorização se solicitado. Verifique que na aba PRESENCAS aparecem as colunas ENTIDADE, TIPO, TIMESTAMP.
  2. 6
    Implantar nova versão
    Implantar → Gerenciar implantações → ✏️ lápis → Nova versão → Implantar. NÃO clique em "Nova implantação".
  3. 7
    Adicionar métodos em js/api.js
    Abra js/api.js no repositório GitHub. Dentro do objeto Api, após getPresencas, cole os 3 métodos do arquivo api_BlocoA_adicoes.js.
  4. 8
    Fazer upload do reuniao.html
    Copie reuniao.html para a raiz do repositório cmsmoc/doc. Faça git push. Aguarde o GitHub Pages atualizar (~30s).
  5. 9
    Ativar a reunião no CONFIG
    Na planilha → aba CONFIG → encontre ou crie a chave reuniao_ativa_id. Preencha com o ID da reunião desejada (da aba REUNIOES). Ou simplesmente certifique-se que a reunião tem STATUS=ativo.

BLOCO B — Deploy do HealthCheck

  1. 1
    Colar o Code_BlocoB_healthcheck.gs no Code.gs
    Logo abaixo do conteúdo do Bloco A, antes da seção de utilitários.
  2. 2
    Integrar o CSS ao Painel.html
    Abra Painel.html no GAS. Localize o bloco <style>. Copie apenas o conteúdo dentro das tags <style id="hc-styles-reference"> do arquivo Painel_BlocoB_healthcheck.html e cole ao final do bloco style existente.
  3. 3
    Adicionar o botão de aba
    Localize a barra de abas do Painel e adicione:
<button class="tab-btn" onclick="mudarAba('healthcheck')">🩺 HealthCheck</button>
  1. 4
    Adicionar o painel HTML
    Copie o <div class="tab-pane" id="tab-healthcheck"> do arquivo Painel_BlocoB_healthcheck.html e cole junto com os outros painéis.
  2. 5
    Adicionar o JavaScript
    Copie o conteúdo dentro das tags <script id="hc-script-reference"> e cole no bloco <script> do Painel.html.
  3. 6
    Salvar e implantar nova versão
    Salvar (Ctrl+S) → Implantar → Gerenciar implantações → ✏️ → Nova versão → Implantar.

Sandbox — Ambiente de Testes

🧪 Estrutura do Sandbox

O sandbox é um ambiente paralelo que usa uma planilha clone e um script GAS separado. O frontend fica em cmsmoc/doc/dev/ ou como branch dev do repositório.

💡
Custo zero. O sandbox usa exatamente a mesma infraestrutura do prod (GitHub Pages + GAS gratuito). Basta duplicar a planilha e criar um novo deploy GAS apontando para ela.

Passo a passo para criar o Sandbox

  1. 1
    Duplicar a planilha
    Planilha → Arquivo → Fazer uma cópia. Nome sugerido: "CMS-MOC DEV". Anote o ID da nova planilha (na URL).
  2. 2
    Criar script GAS para o DEV
    Na planilha dev → Extensões → Apps Script. Cole o Code.gs atual. Em Script Properties, defina SHEET_ID como o ID da planilha dev. Implante como Web App e anote a URL dev.
  3. 3
    Criar config de DEV no frontend
    No repositório, crie o arquivo dev/config/config.js com a URL GAS do DEV. O config.js produção aponta para a URL prod.
// dev/config/config.js
const CONFIG = {
  GAS_URL: 'https://script.google.com/macros/s/SEU_ID_DEV/exec',
  SESSION_KEY: 'cms_session_dev',  // chave diferente para não confundir sessões
  ENV: 'dev'
};
⚠️
O sandbox compartilha o mesmo domínio GitHub Pages que o prod. Para distinguir, adicione um banner visual na dev: if (CONFIG.ENV === 'dev') { document.body.style.outline = '4px solid red'; }

Checklist de Testes pós-Deploy

✅
Execute esses testes em sequência, de preferência em uma reunião de teste (STATUS=ativo na planilha com uma reunião fictícia).

Bloco A — Presenças

  • reuniao.html carrega — Acesse cmsmoc.github.io/doc/reuniao.html. Deve exibir loading screen e depois a página.
  • Sem reunião ativa — Com todas reuniões STATUS≠ativo, deve exibir "Nenhuma reunião ativa no momento".
  • Reunião ativa aparece — Coloque STATUS=ativo em uma reunião. Recarregue. Hero card deve exibir título, data, hora, local.
  • Visitante consegue registrar presença — Clique em "Sou visitante", preencha nome + entidade, clique em registrar. Verifique na planilha aba PRESENCAS se a linha apareceu com TIPO=visitante.
  • Anti-duplicata visitante — Tente registrar o mesmo nome novamente. Deve retornar "Você já está registrado".
  • Login inline funciona — Clique em "Sou conselheiro", faça login com credenciais válidas. Botão de presença deve aparecer.
  • Marcar presença de conselheiro — Logado, clique em "Confirmar minha presença". Toast de sucesso deve aparecer. Botão vira "Presença já confirmada".
  • Idempotência — Tente marcar presença novamente (via URL direta ou atualizando a página). Não deve duplicar na planilha.
  • Barra de presença — Contador e porcentagem devem atualizar conforme presenças são adicionadas.
  • Lista de presenças — Lista deve exibir todos os registrados com avatar de iniciais.
  • Auto-refresh — Aguarde 30 segundos. "Última atualização" deve atualizar o horário.
  • Mobile — Abrir no celular. Interface deve ser usável (botão de presença de fácil toque).

Bloco B — HealthCheck

  • Aba HealthCheck aparece no Painel Admin — Abra o Painel Admin (sidebar da planilha). Botão "🩺 HealthCheck" deve estar visível.
  • Verificação executa — Clique em "▶ Executar verificação". Todos os 12 cards devem sair de "loading" para ok/warn/err.
  • Abas obrigatórias — OK — Card "Abas obrigatórias" deve mostrar ✅ OK se Setup já foi executado.
  • Presenças v3.2 — OK — Card "Presenças v3.2" deve mostrar ✅ após executar _setupPresencasExt.
  • Quotas — percentual visível — Card de quotas deve exibir número de requisições do dia e percentual.
  • Log técnico popula — Área de log técnico deve exibir linhas coloridas com o resultado de cada verificação.
  • Executar novamente — Botão muda para "▶ Executar novamente" e permite re-executar.

Troubleshooting

SintomaCausa provávelSolução
reuniao.html fica em loading infinito GAS não foi implantado ou URL incorreta no config.js Verificar CONFIG.GAS_URL no console do navegador (F12). Confirmar que nova versão foi implantada.
"Ação inválida: getReuniaoCurrent" Cases não foram adicionados ao switch do doGet() Adicionar os 3 cases ao switch e reimplantar nova versão.
Presenças não aparecem na lista getPresencas ainda retorna formato antigo Substituir handleGetPresencas pela versão handleGetPresencas_v32 no Code.gs e renomear para handleGetPresencas.
Erro "Aba PRESENCAS — coluna fora do índice" Colunas ENTIDADE/TIPO/TIMESTAMP não foram criadas Executar _setupPresencasExt() no GAS.
HealthCheck — cards ficam em "Verificando..." para sempre google.script.run não conseguiu chamar a função Verificar no console do Painel (F12) se há erros. As funções hcVerificar* devem estar no Code.gs salvo e implantado.
Login inline não funciona na reuniao.html Api.login não está disponível ou retorna erro Verificar se js/auth.js e js/api.js estão sendo carregados (Network tab no F12). Api.login deve existir.
